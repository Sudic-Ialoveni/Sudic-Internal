API - General Information

Technology

The API allows you to integrate My Calls with an external system (for example, CRM or 1C).

The following options are available:

  • Starting an outgoing call.
  • Sending SMS.
  • Getting a list of calls from the database.
  • Receiving events of call start/answer/end, SMS receiving and sending.

Access to the API is implemented in two versions: REST HTTP and Webhook.

All requests and responses are transmitted in JSON format.

Depending on the tasks and technical capabilities, you can choose one of the options or combine them.

When to use REST API [HTTP request->response model]:

  • You only need to make outgoing calls/sms.
  • You need to periodically (not frequently) receive a list of call history, for example, once an hour/day or when launching a client application.

When to use the Webhook API [incoming POST requests for call/SMS events from our server to yours]:

  • You need to receive call start/answer/end events (or just call history) or SMS events in real time.
  • You have your own dedicated server where you can receive incoming POST requests from our server.
    Calls and SMS messages can then be initiated via the REST API.

REST API

Authorization

To execute REST requests, authorization is required.
Authorization data is passed in each request and consists of the username (email) and API key.

Executing queries

Requests are made using an HTTP POST request with the content type application/json ( Content-Type: application/json header ). Requests specifying a different Content-Type will fail.
The request body contains a JSON array converted to a string. Each request must contain the following fields:

  • user_name - user email for authorization.
  • api_key - API key for authorization, it can be found in your personal account, Settings->Integration.
  • action - the API method to call.

The request parameters themselves are added to this main JSON array.

The URL for the request is as follows: https://[domain].moizvonki.ru/api/v1


For example, a method call to make an outgoing call from the user's phone number user@mail.ru to number 12345678 would look like this (the example is written in Javascript using jQuery):

var requestData = JSON.stringify({
user_name: "user@mail.ru",
api_key: "pempgqsbvq9wzkmy1smvkhvihkkyk5qm",
action: "calls.make_call",
to: "12345678"
});
$.post ("https://test.moizvonki.ru/api/v1", {request_data: requestData});


The HTTP response will contain the result of executing the API method in JSON format.

If an error occurs during the request, an HTTP response code other than 200 OK and a description of the error will be returned in the response body.

Webhook API

The Webhook API is designed to receive call events in real time by sending a POST request from our server to yours.
To start receiving events, you first need to subscribe to them by specifying the URL for receiving events on your side.

Event subscriptions are performed using the REST API .

Please note: these requests can only be made on behalf of the Administrator. Subscriptions are made once for the entire account (i.e., there is no need to do this for each user separately). Thus, events from all phones in the account will be sent to the specified webhook.


Subscribing to events - webhook.subscribe

{[поля авторизации REST,] "action": "webhook.subscribe", "hooks": {"call.start": "https://mysite/myurl", "call.answer": "https://mysite/myurl", "call.finish": "https://mysite/myurl?myparam=value"}}

The hooks array contains a list of events to subscribe to, including:
Key - the event name
; Value - the URL of the handler on your server.

The following events are available:
  • call.start - start of a call
  • call.answer - answer a call
  • call.finish - ends a call
  • sms.message - SMS message (SMS recording must first be enabled in the app settings on your specific phone; it's disabled by default)

* You can subscribe to one specific event (for example, the end of a call to get the call log) or to all of them at once
* You can use the same URL or different ones for each event
* If you call the subscribe method on previously subscribed events, the URL of the handler of this event will be replaced with a new one, i.e. there can be only one handler per event
* The URL must contain a scheme (http:// or https://)
* The URL can contain parameters (?param=value)
* For an outgoing call, call.answer is always sent immediately after call.start, since phones do not transmit the answer time during an outgoing call. The correct answer status of an outgoing call will be transmitted at the end of the call in the call.finish event.
* The call.finish event can be sent without the corresponding call.start. This will happen if the Internet was turned off on the phone during the call and when loading previously made calls, only the call.finish event will be sent (which will contain all the information about the call, including the start and answer times). The call.start/answer event is sent only when a call is processed in real time (phone online).

Unsubscribing from events - webhook.unsubscribe

{[поля авторизации REST,] "action": "webhook.unsubscribe", "hooks": ["call.start", "call.answer", "call.finish"]}

The hooks list contains a list of events to unsubscribe from. Once unsubscribed, your server will stop receiving the specified events.

Get a list of subscriptions - webhook.list

{[поля авторизации REST,] "action": "webhook.list"}

The answer is as follows:

{"call.start": "https://mysite/myurl", "call.answer": "https://mysite/myurl", "call.finish": "https://mysite/myurl?myparam=value"}

In the response, you will receive an array of events to which you have subscribed:
Key - the name of the event
Value - the URL of the handler on your server.

Request format from our server

When an event occurs, a POST request will be sent to your URL with data in JSON format in the request body:

{"webhook": {"action": "call.start", "account_id": "1", "account_name": "test", "user_id": "2", "user_login": "test@mail.ru"}, "event": [данные события]}

Where:

  • webhook - webhook metadata:
    action - event name
    account_id - account ID where the event occurred
    account_name - account name where the event occurred
    user_id - user ID that generated the event (e.g. made a call)
    user_login - login (email) of this user
  • event - event data.

The contents of the event field are the event data.

All call events contain the following required fields:

  • event_type - event type: 1 - call start; 2 - call answer; 4 - call end; 32 - SMS.
  • event_created - time of event creation on the server (UTC).
  • direction - call/SMS direction: 0 - incoming, 1 - outgoing.
  • client_number - client (subscriber) phone number.
  • client_name - the client's name, if the client is in the My Calls contact list, or the contact name from the phone's phone book.
  • event_pbx_call_id - temporary call ID. All events related to the same call will have the same ID.

Call start and answer event

The event data does not contain any additional fields.

Please note that these events may not always appear. For example, if your phone only uses mobile data, the internet connection may be disconnected during a call (depending on the network), and therefore the phone will not be able to notify you when the call is started or answered. In this case, you will only receive the call end event.

Call end event

The event contains the following additional fields:

  • start_time - call start time, UTC timestamp.
  • answer_time - the time the call was answered, UTC timestamp. If the call was not answered, 0.
  • end_time - call end time, UTC timestamp.
  • duration - call duration in seconds.
  • answered - 0 if the call was not answered and 1 if it was answered.
  • recording - a link to the recording of the conversation if the call is answered.
  • db_call_id - Call ID in the database.

SMS receiving/sending event

The event contains the following additional fields:

  • text - message text.


Please note that these events may appear after the actual call/SMS end. This is because the phone app may be running offline. This means that when the phone is not connected to the internet, calls/SMS are stored in the phone's local database. When connected to the internet, the data will be uploaded to the server and then sent via webhook. Also, in this case, call start/answer events will not be transmitted ! If you receive a call end event without a start, add it to the database for this event; this is normal behavior.
If the phone is connected to the internet, events are sent in real time.

API Functions - Making a Call/Sending SMS

Make an outgoing call - calls.make_call

To make an outgoing call via a user's smartphone, send a request of the following type:

{[поля авторизации, в случае REST,] "action": "calls.make_call", "to": "номер телефона"}

The dialing will be done from the phone of the user under whose login the request is made.


Send SMS - calls.send_sms

Please note that after initiating an SMS message, the user must confirm this action on their smartphone by default. You can choose to send SMS messages without confirmation in our app's settings on your phone (see the "SMS Sending Mode" setting).

To send an SMS via a user's smartphone, send a request of the following type:

{[поля авторизации, в случае REST,] "action": "calls.send_sms", "to": "номер телефона", "text": "текст сообщения"}

The SMS will be sent from the phone of the user under whose login the request is made.


Get a list of SMS templates - calls.get_sms_templates

You can get a list of SMS templates that are set in your personal account My Calls (Settings->Calls and SMS->SMS templates).

To obtain templates, send a request of the following type:

{[поля авторизации, в случае REST,] "action": "calls.get_sms_templates"}

The answer is as follows:

{"results_count": 1, "results": [{"text": "Здравствуйте, Ваш заказ готов!", "description": "Информирование о готовности заказа", "id": 1, "name": "Заказ готов"}]}

Where:

  • results_count - the number of templates in the response.
  • results - list of templates.

API Functions - Getting a Call List

Get a list of calls from the database - calls.list

The request looks like this

{[поля авторизации,] "action":"calls.list", "from_date": 1422774918, "from_id": 20, "max_results": 5, "to_date": 1422784918, "from_offset": 798, "supervised" : 0}

Where:

  • from_date - a required parameter (if from_id is not specified), specifies the date from which to retrieve calls, in UTC timestamp. Do not use this parameter for a sequential call list; use from_id for that purpose.
  • from_id - a required parameter (if from_date is not specified), specifies the call ID after which calls should be retrieved (including from_id). Using from_id, you can retrieve a list of calls sequentially by making a request with a from_id greater than the ID of the last received call (the db_call_id field in the call details + 1). Specifying 0 or 1 will retrieve the entire list of calls.
    If both from_id and from_date are specified, only from_id is used.
  • to_date - optional parameter, specifies the date for which to receive calls, UTC timestamp.
  • from_offset - an optional parameter that specifies the starting index in the selection. Since the size of the call list in the response is limited, use this parameter to retrieve the entire list sequentially.
  • max_results - the maximum number of results returned, valid value 1 - 100.
  • Supervised is an optional parameter. If 1
    is specified , all calls available to this user based on their role (Administrator, Manager, etc.) will be returned. If 0 is specified or the attribute is not set, only calls from this user will be returned.

The result of executing the method will return a JSON array with call data containing the following fields:

  • direction - call direction: 0 - incoming, 1 - outgoing.
  • client_number - client (subscriber) phone number.
  • client_name - the client's name, if the client is in the My Calls contact list, or the contact name from the phone's phone book.
  • start_time - call start time, UTC timestamp.
  • upload_time - time of call upload from phone to server, UTC timestamp.
  • answer_time - the time the call was answered, UTC timestamp. If the call was not answered, 0.
  • end_time - call end time, UTC timestamp.
  • duration - call duration in seconds.
  • answered - 0 if the call was not answered and 1 if it was answered.
  • recording - a link to the recording of the conversation if the call is answered.
  • db_call_id - Call ID in the database.
  • src_number - the phone number of the SIM card through which the call was made, if the SIM card numbers are specified in the Personal Account and the SIM card is identified in the call.
  • src_id - SIM card ID in the database.
  • src_slot - SIM card slot.

The results_next_offset attribute is also returned in the response . If there are still results to return in the selection, use this value as the from_offset parameter in the subsequent request to retrieve the entire list sequentially. If the supervised=1
parameter is specified in the request , information about the call owner will be returned in the data for each call:

  • user_account - user account (email).
  • user_id - user identifier in the system.


Example answer:

{ "results_next_offset": 0, "results_count": 1, "results": [ { "direction": 1, "user_account": "test@test.ru", "client_number": "+7144446272", "start_time": 1495627000, "src_number": "+7442446333", "duration": 10, "user_id": 1, "answer_time": 1495627000, "upload_time": 1565958191, "src_id": 8, "client_name": "", "src_slot": 1, "recording": "", "answered": 1, "db_call_id": 1224, "event_pbx_call_id": "1.1495627000", "end_time": 1495627010 } ], "results_remains": 0 }

API Functions - Account Management

Please note: These requests can only be made on behalf of the Account Administrator.


Add an employee - company.create_employee

The request looks like this:

{[поля авторизации,] "action":"company.create_employee", "employee_user_name":"mail@mail.ru", "employee_password": "password", "employee_display_name": "Василий Иванов", "employee_role": 4, "employee_group_id": 1}

Where:

  • employee_user_name - email (login) of the new user
  • employee_password - new user password (minimum 6 characters)
  • employee_display_name - the name of the new user
  • employee_role - optional parameter, role of the new user: 2 - manager, 3 - group leader, 4 - manager (default)
  • employee_group_id - optional parameter, ID of the group to which the new user will be added (the primary group by default)

The response will return the ID of the created user.


Delete an employee - company.delete_employee

Deletes the user and all their data (calls and contacts). This action is irreversible!
The request looks like this:

{[поля авторизации,] "action":"company.delete_employee", "employee_user_name":"mail@mail.ru"}

Where:

  • employee_user_name - email (login) of the user being deleted
    OR employee_id - ID of the user being deleted (you can pass either employee_user_name or employee_id as desired)

Activate/Deactivate an employee - company.activate_employee

The request looks like this:

{[поля авторизации,] "action":"company.activate_employee", "employee_user_name":"mail@mail.ru", "activate": false}

Where:

  • employee_user_name - email (login) of the user being changed
    OR employee_id - ID of the user being changed (you can pass either employee_user_name or employee_id as desired)
  • activate - if false, the user is deactivated, true - activated

Get a list of users - company.list_employee

The request looks like this:

{[поля авторизации,] "action":"company.list_employee", "max_results": 10, "from_offset": 0}

Where:

  • from_offset - an optional parameter that specifies the starting index in the selection. Since the size of the call list in the response is limited, use this parameter to retrieve the entire list sequentially.
  • max_results - optional parameter (default 100), maximum number of results returned, valid value 1 - 100.
  • employee_user_name - optional parameter, email (login) of the user, if specified, then information will be returned only for this user
    OR employee_id - user ID (you can pass either employee_user_name or employee_id as desired)

The answer is as follows:

{"results_count": 1, "results_remains": 0, "results_next_offset": 0, "results": [{"display_name": "Василий Петров", "email": "mail@mail.ru", "group_name": "Отдел продаж", "group_id": 1, "role": 4, "id":1}]}


Get a list of groups - company.list_group

The request looks like this:

{[поля авторизации,] "action":"company.list_group", "max_results": 10, "from_offset": 0}

Where:

  • from_offset - an optional parameter that specifies the starting index in the selection. Since the size of the call list in the response is limited, use this parameter to retrieve the entire list sequentially.
  • max_results - optional parameter (default 100), maximum number of results returned, valid value 1 - 100.

The answer is as follows:

{"results_count": 1, "results_remains": 0, "results_next_offset": 0, "results": [{"name": "Отдел продаж", "description": "Эти люди продают", "id":1}]}
